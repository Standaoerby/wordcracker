# HOWTO_REMEDIATION — прогнать доработки через Claude Code

> Операционная инструкция к `REMEDIATION_BRIEF.md`. Пять задач
> (T0 → T1 → T2 → T4 → T5), по одной за сессию, каждый гейт проверяешь
> сам. Главное правило: задача закрыта не когда Claude Code сказал
> «готово», а когда **ты лично** прогнал команды гейта и увидел зелёное.

---

## 0. Подготовка — один раз перед стартом

```bash
# 0.1 — снять зависший git-лок (иначе Claude Code не закоммитит)
del C:\_PROJECTS\wordcracker-code\.git\index.lock        # cmd / PowerShell
# или в Git Bash:  rm -f /c/_PROJECTS/wordcracker-code/.git/index.lock

# 0.2 — закоммитить брифы, чтобы сессии стартовали с чистого дерева
cd C:\_PROJECTS\wordcracker-code
git status                       # убедись, что дерево понятное
git add REMEDIATION_BRIEF.md HOWTO_REMEDIATION.md
git commit -m "docs: REMEDIATION + HOWTO — доводка фаз 0-6"

# 0.3 — проверить, что тесты вообще запускаются у тебя локально
python -m pytest tests/v2 --collect-only -q | tail -3
# ожидаемо: «1467 tests collected», 0 ошибок коллекции
```

`CLAUDE.md` уже в корне — Claude Code подхватывает его автоматически
каждой сессией, настраивать ничего не нужно.

Если `pytest` не находит зависимости — поставь их в своё dev-окружение
(`pip install -r requirements*.txt` или как заведено в проекте). Сьют
рассчитан на запуск и на Windows тоже (в нём есть `WindowsDevPath`-тесты).

## 1. Как устроен один цикл

Для каждой задачи T# повторяется одно и то же:

1. Открыть в VS Code интегрированный терминал в `C:\_PROJECTS\wordcracker-code`.
2. Запустить `claude`.
3. Вставить промпт задачи (раздел ниже). Дать доработать.
4. **Самому** прогнать команды «Проверка гейта» — в отдельном терминале.
5. Зелёный гейт → задача принята. Красный → вернуть Claude Code с
   конкретным пунктом, что не сошлось.
6. Проверить `git log --oneline -5`: коммитов по задаче немного, без
   каскада (R7 — 3+ коммита в одну область = стоп).
7. `/clear` или перезапуск `claude` — и следующая задача.

**Порядок незыблем:** T0 → T1 → T2 → T4 → T5. Не брать T1, пока гейт
T0 не сошёлся лично у тебя.

## 2. Окружение для проверок

Команды проверок написаны под **Git Bash** (ставится вместе с Git for
Windows — у тебя он есть, раз это git-репозиторий). В VS Code: выпадашка
терминала → «Git Bash».

`pytest` работает в любом терминале. `grep`/`wc` — только в Git Bash или
WSL. Где это критично, ниже дан и вариант для PowerShell.

## 3. T0 — Фаза 0: сьют зелёный и независимый от порядка

### Промпт для Claude Code

```
Прочитай CLAUDE.md и REMEDIATION_BRIEF.md в корне репозитория.
Выполни ТОЛЬКО задачу T0 (Фаза 0). Не начинай T1 и последующие.
Соблюдай R1–R10 из CLAUDE.md. Один фикс = один коммит (R7).

Контекст по T0 уже установлен в брифе и его не надо переоткрывать:
test_q15_compare_empty и test_scoring_plugins падают ТОЛЬКО в составе
сьюта, в одиночку проходят — это загрязнение глобального состояния
чужим тестом. Чини протекающий тест, а не эти две жертвы.

Когда считаешь, что готово:
1) прогони полный `python -m pytest tests/v2` И `python -m pytest
   tests/v2 -p no:randomly` — покажи мне обе сводки;
2) пройди по чеклисту «Gate T0» из брифа пункт за пунктом;
3) если хоть один пункт не зелёный — НЕ коммить, доложи блокер.
```

### Проверка гейта (делаешь сам, Git Bash)

```bash
cd C:\_PROJECTS\wordcracker-code
python -m pytest tests/v2 -q                 # → 0 failed
python -m pytest tests/v2 -q -p no:randomly  # → тот же результат
# выборочно: тест-жертвы и в одиночку, и в сьюте дают один вердикт
python -m pytest tests/v2/test_q15_compare_empty.py tests/v2/test_scoring_plugins.py -q
```

### Зелёный гейт T0

- `pytest tests/v2` → **0 failed**; skip — только с причиной в самом тесте.
- Прогон с `-p no:randomly` и без — одинаковый результат.
- Падение `test_tool_migration` закрыто моком v1 или `@skipUnless` (НЕ
  заменой golden-числа 123 на `assertGreater(0)`).

## 4. T1 — Фаза 1: один резолвер + решение по движку

### Промпт для Claude Code

```
Прочитай CLAUDE.md и REMEDIATION_BRIEF.md. Выполни ТОЛЬКО задачу T1
(Фаза 1). Не начинай T2. Соблюдай R1–R10, один фикс = один коммит.

Сначала R8 — подтверди живой путь: какое значение WC_DEFAULT_ENGINE
реально в проде и какой резолвер/исполнитель реально вызывается.
Зафиксируй вывод в docs/v2/decisions.md.

Если окажется, что прод бегает на движке v1 — ОСТАНОВИСЬ и доложи мне,
не продолжай: это меняет смысл всей доводки.

Используй ИСПРАВЛЕННЫЙ гейт Фазы 1 из Части 3 брифа (буквальный
grep ≤1 — ошибочная формулировка, конфиг-переменные удалять не надо).

Когда готово: прогони полный `python -m pytest tests/v2`, пройди по
чеклисту «Gate T1» из брифа. Красный гейт — не коммить, доложи.
```

### Проверка гейта (делаешь сам, Git Bash)

```bash
# один резолвер — у удалённого не осталось живых импортов
grep -rn "entity_resolver" scripts/ --include=*.py | grep -v __pycache__
# ожидаемо: ссылки только на ОДИН резолвер (v6 либо старый — что выбрали)

# нет флага поколения/мёртвой ветки (WC_V5_*/WC_V6_* — только в комментах)
grep -rnE "os\.(environ\.get|getenv)\(['\"]WC_V[56]" scripts/ --include=*.py
# ожидаемо: пусто

# решение по движку и тумблерам записано
grep -n "WC_DEFAULT_ENGINE\|WC_CRITIC\|WC_NUMERIC_AUDIT" docs/v2/decisions.md

python -m pytest tests/v2 -q                 # → 0 failed
```

### Зелёный гейт T1

- Один файл/пакет резолвера; у удалённого — ноль живых импортов.
- Один исполнитель роутера + один stream-вариант.
- Ни одна env-переменная не выбирает поколение и не гейтит мёртвую ветку.
- Решения по движку и по `WC_CRITIC`/`WC_NUMERIC_AUDIT` — в `docs/v2/decisions.md`.
- `pytest tests/v2` → 0 failed.

## 5. T2 — Фаза 2: убрать цепочки `.get() or .get()`

### Промпт для Claude Code

```
Прочитай CLAUDE.md и REMEDIATION_BRIEF.md. Выполни ТОЛЬКО задачу T2
(Фаза 2). Не начинай T4. Соблюдай R1–R10, один фикс = один коммит.

Различай два вида цепочек (см. T2 в брифе): вид 1 — угадывание ключа
v1 (чинится контрактом, R3); вид 2 — нормализация внутреннего dict v2
(чинится одной точкой нормализации). НЕ просто схлопывай `or`.
Если правишь мок под реальный v1 — это R4: чини мок, не враппер.

Когда готово: прогони полный `python -m pytest tests/v2`, пройди по
чеклисту «Gate T2». Красный гейт — не коммить, доложи.
```

### Проверка гейта (делаешь сам, Git Bash)

```bash
# ноль .get()-or цепочек в tools/**
grep -rnE "\.get\([^)]*\)[[:space:]]*or[[:space:]]+[a-zA-Z_.]*\.get\(" scripts/v2/tools/
# ожидаемо: ПУСТО

python -m pytest tests/v2 -q                 # → 0 failed
```

PowerShell-вариант первой проверки (если нет Git Bash):
```powershell
Get-ChildItem scripts\v2\tools -Recurse -Filter *.py |
  Select-String -Pattern '\.get\([^)]*\)\s*or\s+[A-Za-z_.]*\.get\('
# ожидаемо: ничего не выведено
```

### Зелёный гейт T2

- Grep по `.get()-or` в `scripts/v2/tools/` — ноль совпадений.
- Каждый враппер, читающий результат v1, — за `@v1_contract`; удаление
  ключа из схемы v1 роняет ровно один контракт-тест.
- `pytest tests/v2` → 0 failed.

## 6. T4 — Фаза 4: декомпозиция `plan.py`

> Самая длинная задача. Инвариант fan-out уже вынесен — это чистый
> рефактор-перенос, поведение меняться не должно.

### Промпт для Claude Code

```
Прочитай CLAUDE.md и REMEDIATION_BRIEF.md. Выполни ТОЛЬКО задачу T4
(Фаза 4). Не начинай T5. Соблюдай R1–R10.

Это чистый рефактор-перенос: вынести _plan_*-builder'ы из plan.py в
пакет scripts/v2/planner/builders/ по доменам, PLAN_BUILDERS сделать
статическим реестром. Поведение НЕ меняется — до и после прогон
pytest tests/v2 даёт идентичный результат. R6: не тащи инлайн-копии
fan-out, зови apply_fan_out_invariant.

Если контекст сессии заканчивается — сделай /compact и продолжи, не
бросай задачу на полпути.

Когда готово: прогони полный `python -m pytest tests/v2`, пройди по
чеклисту «Gate T4». Красный гейт — не коммить, доложи.
```


## 9. Если что-то пошло не так

| Симптом | Что делать |
|---|---|
| Claude Code хочет перейти к следующей T# | Останови. «Одна сессия = одна задача» — промпт это запрещает. |
| Хочет закоммитить при красном гейте | Не давай. Гейт зелёный = коммит. Верни с конкретным красным пунктом. |
| 3-й коммит подряд в одну область | R7: это структурная проблема. Останови, пусть эскалирует, не продолжает каскад. |
| Кончается контекст на T4 | `/compact`, продолжить. Не бросать задачу на полпути. |
| Правит `test_q15`/`test_scoring_plugins` напрямую | Неверно — это жертвы. Чинить надо протекающий тест (T0). |
| `pytest` зелёный у Claude Code, красный у тебя | Сверьте окружение/зависимости; верный вердикт — твой локальный прогон. |
| Не может закоммитить, `index.lock` | Удали `.git\index.lock` (шаг 0.1) и повтори. |
| Прод оказался на движке v1 (всплыло в T1) | Стоп. Эскалация — это меняет смысл доводки, реши до продолжения. |
